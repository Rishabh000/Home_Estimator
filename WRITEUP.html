Spark Homes — Repair Cost Estimator

One-page writeup

1. Most interesting design / UX decision

I built the app around a flexible data model instead of fixed screens. A catalog of the 108 priced items, a set of group templates (the 19 reference groups), and room types that reference those groups. Rooms are instances an agent adds or removes on the fly, and bathrooms/bedrooms/living areas auto-number. This directly serves the brief's most-emphasized requirement, that every house is different, and it kept the code small.

The other decision I care about: totals and the progress bar are patched into the DOM in place on every quantity or checkbox change rather than re-rendering the list. On a phone in a real walkthrough, that means no flicker and no lost keyboard focus while typing a quantity. The interface is a dark, high-contrast, thumb-reachable layout (bottom nav, big tap targets, 16px inputs to stop iOS zoom) tuned for a dim, empty house rather than a screenshot.

2. What is broken or fragile

  • AI Smart Add depends on a model download. The TensorFlow.js sentence encoder (~25 MB) loads on first use and is then cached for offline use. Until it loads, Smart Add falls back to keyword matching, so it degrades gracefully but the first semantic search needs a network connection. Embedding quality on very short item names is good but not perfect.
  • localStorage is the only store. Photos are downscaled to ~1280px JPEGs to fit, but a project with dozens of photos can approach the browser's ~5MB quota. A production version would move photos to IndexedDB.
  • Item→group mapping is my interpretation. The CSV has no group column, so I assigned each item to the most sensible group and added three helper groups (General/Misc, Lighting, Closet) to keep every item reachable and to decouple Living/Common from Interior as the brief requested.
  • No multi-device sync (by design — fully offline, no backend).

3. Creative addition — Deal Analyzer + AI Smart Add

The estimate answers "what will repairs cost." The job is really "should we make this offer?" The Deal Analyzer tab pulls the live repair total, applies the investor 70% rule to suggest a Maximum Allowable Offer, and projects net profit and return on cost after holding and selling costs, ending in a clear Go / Caution / Stop verdict. It converts the agent's checklist into the actual decision they are standing in the house to make, reusing data already captured so it costs zero extra input.

My AI feature is Smart Add: the agent types a problem in plain English ("the AC outside is dead", "carpet is ruined") and an on-device neural model (TensorFlow.js Universal Sentence Encoder) embeds the phrase and ranks the closest of all 108 line items, then one tap drops each into the correct room and group. It is genuine semantic matching, not keyword search, so "AC unit outside" surfaces Condensing Unit even though those words never appear in the item name. I deliberately chose this over photo OCR / object detection: agents work in empty distressed houses where a vision model often has little to see, whereas plain-language lookup speeds up the core task in any house. The model lazy-loads on first use (keeping startup fast) and falls back to keyword search when offline, so it never feels broken.

4. What I'd ship next with two more days

  • Move photo storage to IndexedDB and add per-line-item photo attachment.
  • Project comparison view and a duplicate-project / template feature for similar homes.
  • Square-footage helper that auto-fills quantities for area-based items (paint, flooring).
  • Sharable read-only estimate link / PDF export alongside the Excel ZIP.
  • Voice input into AI Smart Add, and pre-bundling a smaller quantized embedding model so the first semantic search works without any network.

Role of AI tools

I used an AI coding assistant to accelerate scaffolding, the CSV-to-data-model transcription, and boilerplate (service worker, export plumbing). All architecture decisions, the data model, the UX, the Deal Analyzer concept, the AI Smart Add design, and final review/testing were mine; AI was a pair-programmer, not the author of the design.
